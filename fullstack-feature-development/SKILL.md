# 全栈功能开发生命周期管理

## 使用场景矩阵

| 场景 | 触发语句 | 使用的检查点 | 产出物 |
|------|---------|-------------|--------|
| **新功能开发** | "开发xxx功能" | CP0→CP1→CP2→CP3→CP4→CP5 | 完整功能+检查清单 |
| **功能验证** | "验证xxx功能" | CP1→CP2→CP3→CP4→CP5 | 验证报告 |
| **Code Review** | "审查xxx代码" | CP1→CP2→CP3→CP4 | 审查意见 |
| **上线前检查** | "准备上线" | CP1→CP2→CP5+部署检查 | 上线检查清单 |
| **Bug修复** | "修复xxx问题" | CP0(根因)→修改→CP5 | 修复报告 |

## 5个核心检查点（CP）详解

每个检查点有：**适用场景** + **通过标准** + **验证方法**

### CP0: 需求/设计确认
- **适用场景**：新功能开发、Bug修复（根因分析）
- **通过标准**：业务逻辑清晰，边界明确
- **验证方法**：AI提问，用户确认

### CP1: 接口/路由存在
- **适用场景**：全部场景
- **通过标准**：新功能的路由/接口已注册
- **验证方法**：
  ```bash
  # 根据项目类型自动选择
  grep "路由名" backend/src/app.js      # Express
  grep "路由名" backend/urls.py         # Django
  grep "路由名" src/app.ts              # NestJS
  ```

### CP2: API契约合规
- **适用场景**：全部场景
- **通过标准**：响应格式符合 {code, data, message}
- **验证方法**：
  ```bash
  curl -X [METHOD] [URL] [参数] | jq
  ```

### CP3: 数据访问正确
- **适用场景**：涉及数据库的功能
- **通过标准**：使用正确的数据库连接，无常见错误
- **验证方法**：
  ```bash
  # 项目特定的检查
  grep -r "getPool()" | grep "users"    # 你的项目
  grep -r "SQL注入风险"                 # 通用检查
  ```

### CP4: 前端数据流正确
- **适用场景**：涉及前端的功能
- **通过标准**：正确调用API，正确处理响应
- **验证方法**：
  ```bash
  # 项目特定的检查
  grep "response.code === 200" frontend/src/store/
  grep "api.xxx(" frontend/src/store/   # 是否调用了新API
  ```

### CP5: 端到端可用
- **适用场景**：全部场景（除了纯Code Review）
- **通过标准**：用户操作能完成完整流程
- **验证方法**：
  - 功能验证：人工测试关键路径
  - 上线前：自动化测试+手动验证
  - Bug修复：回归测试

## 各场景的详细流程

### 场景A：新功能开发（完整流程）

```
触发："开发订单导出功能"

Phase 0: 设计（CP0）
  AI: "确认需求：导出哪些字段？谁可以用？"
  你: 回答确认
  
Phase 1: 后端开发
  AI开发 → 自动验证 CP1/CP2/CP3
  
Phase 2: 前端开发
  AI开发 → 自动验证 CP4
  
Phase 3: 集成验证
  AI测试 → 验证 CP5
  
Phase 4: 交付
  AI: "全部检查点通过，确认交付？"
  你: 确认

产出：完整功能 + docs/checklist-订单导出.md
```

### 场景B：功能验证（你的情况）

```
触发："验证查询历史功能" 或 "检查SQLite缓存是否完成"

Phase 1: 基础检查（AI自动）
  ✅ CP1: 路由注册检查
     grep "query-history" backend-node/src/app.js
     
  ✅ CP2: API契约检查
     curl /api/query-history | jq
     
  ✅ CP3: 数据库连接检查
     grep -r "getPool()" | grep "users"
     
  ✅ CP4: 前端响应检查
     grep "response.code" frontend-pc/src/store/queryCache.js

Phase 2: 端到端验证（AI引导+你执行）
  ⏳ CP5: 功能可用性检查
     
     AI: "请执行以下测试步骤："
         1. 打开预收款统计页面
         2. 点击"异步查询(带缓存)"按钮
         3. 等待查询完成
         4. 查看查询历史页面
         5. 确认历史记录显示正确
     
     你: [执行测试并反馈]
     
     AI: "记录结果到检查清单"

产出：docs/verification-report-查询历史.md
      （包含每个CP的验证结果和证据）
```

### 场景C：Code Review

```
触发："审查用户管理功能的代码"

Phase 1: 静态检查（AI自动）
  ✅ CP1: 路由注册 - 检查路由是否存在
  ✅ CP2: API契约 - 检查响应格式
  ✅ CP3: 数据库 - 检查连接使用
  ✅ CP4: 前端 - 检查响应处理
  
Phase 2: 代码质量检查（AI自动）
  - 是否有重复代码？
  - 错误处理是否完善？
  - 日志是否添加？
  
Phase 3: 生成审查报告
  产出：code-review-用户管理.md
        （包含发现的问题和建议）

注意：Code Review 通常不执行 CP5（端到端测试）
```

### 场景D：上线前最终检查

```
触发："准备上线" 或 "发布前检查"

Phase 1: 功能检查（CP1→CP5）
  对本次上线的所有功能执行检查
  
Phase 2: 部署检查（新增）
  ⬜ 环境变量配置
  ⬜ 数据库迁移脚本
  ⬜ 日志目录权限
  ⬜ 健康检查接口
  
Phase 3: 回滚准备
  ⬜ 回滚方案
  ⬜ 数据备份

产出：docs/pre-launch-checklist.md
```

### 场景E：Bug修复

```
触发："修复查询历史显示问题"

Phase 0: 根因分析（CP0变种）
  AI: "问题是什么？复现步骤？根因？"
  你: 描述问题
  
Phase 1: 修复
  AI修复代码
  
Phase 2: 验证
  ⬜ 修复CP5（回归测试）
     - 原问题是否解决？
     - 是否引入新问题？

产出：docs/bugfix-report.md
      （包含根因、修复方案、验证结果）
```

## 如何触发不同场景

### 功能验证场景

你可以这样说：

```
"验证一下SQLite缓存功能"
"检查查询历史功能是否完成"
"验收一下异步查询功能"
"帮我测试订单导出功能"
```

AI会：
1. 识别这是验证场景（不是开发）
2. 跳过CP0（设计已确定）
3. 执行CP1→CP5
4. 生成验证报告

### Code Review场景

```
"Code Review用户管理功能"
"审查一下这个PR"
"检查查询缓存代码质量"
```

AI会：
1. 识别这是审查场景
2. 执行CP1→CP4（静态检查）
3. 执行代码质量检查
4. 跳过CP5（运行测试）
5. 生成审查报告

### 上线前检查场景

```
"准备上线"
"发布前检查"
"生产环境准备"
```

AI会：
1. 识别这是上线场景
2. 执行全部CP检查
3. 执行部署特定检查
4. 生成上线检查清单

## 总结

这个技能**不只是开发工具**，而是**全生命周期管理工具**：

| 阶段 | 你能说的话 | AI会做什么 |
|------|----------|-----------|
| 开发前 | "设计用户管理功能" | 头脑风暴+CP0 |
| 开发中 | "继续开发" | 编码+CP1-CP4 |
| 开发后 | "验证功能" | CP1-CP5+报告 |
| 审查时 | "Code Review" | CP1-CP4+质量检查 |
| 上线前 | "准备上线" | 全CP+部署检查 |
| 修复时 | "修复bug" | 根因分析+CP5 |

**关键**：根据你的指令，AI自动判断场景，选择对应的检查点组合。
